## Problem Statement

MNEMEX is the third leg of the STRATHEX ecosystem alongside **STRATHEX** (the tournament management platform that captures live results during events) and **STRATHMARK** (the prediction and handicap engine that consumes results to compute marks). MNEMEX's job is to be the **historical memory** of Timbersports: a continuously growing archive of competition results from federations, events, and scorebooks across decades. It feeds STRATHMARK's training data and gives the platform deep historical context that no live scoring system can produce on its own.

```
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│  STRATHEX        │  │  STRATHMARK      │  │  MNEMEX (this)   │
│  Live tournament │  │  Handicap +      │  │  Historical      │
│  management.     │──│  prediction      │←─│  archive.        │
│  Captures        │  │  engine.         │  │  Scrapes the     │
│  current-era     │  │  Computes marks  │  │  past + accepts  │
│  results during  │  │  from historical │  │  federation      │
│  events.         │  │  results.        │  │  backfill.       │
└──────────────────┘  └──────────────────┘  └──────────────────┘
   live results          predictions          historical results
        │                                              ▲
        └──────── all flow into STRATHMARK ────────────┘
                  via the shared HistoricalResult schema
```

Today the historical archive is starved. The STIHL official site has structured pro-circuit data, but it's only a sliver of the sport. The rest lives in three places:

1. **Other federations** (AWFC college conclaves, regional events, lumberjack world championships, ALA/CANLOG association shows) — published as one-off Excel scorebooks per host, with non-standard layouts.
2. **Photographs of paper scoresheets** — Arden Cogar Jr. and other documentarians regularly post pictures of printed match results to social media. Rich data, trapped in images.
3. **Federation internal records** — decades of results that were never published online. ALA, AWFC member schools, and others sit on archives nobody else can access.

Without a way to capture this data reliably, STRATHMARK is starved of training depth, athlete career stats are split across name variants and federations, and the sport itself has no canonical historical record.

## Scope Discipline (load-bearing — added by /autoplan CEO challenge)

MNEMEX is **historical-only**. The discipline is: **scrape first, accept federation historical backfill second, and stop.**

### What MNEMEX is

- **Primary function: scraping.** Collects old results from public sources — federation websites, archived event pages, PDF scorebooks, association newsletters, news coverage, photographed scoresheets. Most of the value is data that exists somewhere online but is unstructured, scattered, or buried in formats nobody has parsed.
- **Secondary function: federation backfill.** Some federations have decades of internal records that were never published. For those, MNEMEX accepts batched uploads from federation administrators (CSV, Excel, structured upload form). **One-way intake for historical data only.**

### What MNEMEX is NOT

- **NOT a portal for federations to enter live or current match results.** That is STRATHEX's job, not MNEMEX's.
- **NOT a scoring system.** MNEMEX never decides who won an event.
- **NOT a competitor to STRATHEX's tournament management.** Federations that want their live results in the ecosystem run their events on STRATHEX, which captures everything automatically through the scoring pipeline.
- **NOT a real-time data source.** MNEMEX data is always at least one event-cycle behind reality, by design.

A parallel live-entry portal in MNEMEX would duplicate STRATHEX's core function, fragment the data pipeline, and create two sources of truth for current-era results. It is forbidden by design.

### Implementation rule

If a feature would let a federation operator enter the result of an event that happened today, it belongs in STRATHEX. MNEMEX features must require either (a) the event happened pre-STRATHEX-adoption-by-that-federation, or (b) the data is being backfilled from an existing archive that the federation already maintains. The `mnemex publish` historical-backfill flow (added by /autoplan CEO challenge) MUST refuse to ingest events with dates within the federation's STRATHEX-active window.

## What Makes This Cool

MNEMEX is the missing data ingest layer. It does three things no current tool does:

1. **Cross-source unification.** STIHL pro circuit + AWFC college conclaves + ALA / CANLOG association newsletters + image-extracted scoresheets + manual paste-in, all funneled into one canonical dataset with athlete-identity dedup across sources.
2. **Vision-LLM extraction with human verification.** Photos of printed scoresheets, association newsletters (PDF / email), and arbitrary Excel files become structured rows. Every row tagged with provenance and verification status. No unverified row reaches the handicap engine.
3. **Two surfaces from the same data.** A web archive (Baseball-Reference-style, hardcore-fan-grade) for browsing — and a STRATHMARK adapter that feeds the handicap engine for tournament tooling.

The "whoa" moment is when a hardcore fan opens a profile page and sees a single athlete's full cross-circuit history — STIHL pro events, AWFC college years, ALA / CANLOG association shows, regional matches, and Arden Cogar's photographed scoresheets — unified for the first time anywhere.

The ALA Sept 2025 newsletter front-page note demonstrates exactly why this matters: Erin LaVoie, Nate Hodges, and Lauren Bergman all set records at the US STIHL Championships and all three are ALA members whose smaller-show results live in newsletters that the STIHL site never references. Today nobody can tell you Erin's full-season story across both circuits. MNEMEX is the place where you can.

## Constraints

- **Score accuracy is non-negotiable.** Inherited from STRATHMARK. Wrong ingest data corrupts handicap math, which makes races unfair. No row reaches the STRATHMARK adapter unverified.
- **Multi-source from day one.** v1 ships with five live ingest paths: STIHL HTML scraper, college Excel via Vision LLM, ALA / CANLOG newsletter PDF via Vision LLM, image ingest via Vision LLM, manual CSV paste-in.
- **Identity dedup is load-bearing**, not a side feature. Cross-source canonicalization has its own service, store, review queue, and audit trail.
- **STRATHMARK schema compatibility is a contract.** A CI test imports STRATHMARK's `HistoricalResult` and validates that MNEMEX's Tier 1 export shape conforms.
- **Style: lean and simple, no unnecessary complexity.** Inherited from the STRATHMARK ecosystem ethos.
- **Output: plain text, no emojis, no ANSI color codes** for any CLI tooling. Inherited from STRATHMARK.

## Premises

1. The aggregator's primary purpose is to be MNEMEX — the canonical "outside-world" data feeder for STRATHMARK and a public archive for hardcore fans/federations.
2. Repo stays separate from STRATHMARK. Schema integration is via library import (`from strathmark.predictor import HistoricalResult`); deployment integration in v1 is JSONL → STRATHMARK's `import_legacy.py`. HTTP POST and shared-Supabase paths are designed and stubbed but **not implemented in v1** — they belong to v1.1+ when live-tournament scenarios drive the work and STRATHEX requires real-time ingest.
3. JSONL → import_legacy.py is the v1 integration path. v1 ships exactly one path; this removes auth surface, retry/idempotency surface, and integration-test cost from the critical path.
4. v1 image ingest is core. Vision LLM extraction + human review queue are first-class v1 components, not v2.
5. Excel format survives as one of N export options, not as the canonical schema.
6. Multi-source schema with `source` and `source_native_id` fields baked in from day 1. Adding a new source = adding a parser, not a schema migration.
7. Quality field is imputed when missing, with provenance flag (`quality_imputed=true`). STRATHMARK's predictor decides what to do.
8. Athlete identity dedup is a first-class service. Stable STIHL `/Athlete/{id}` URLs seed the canonical mapping; image-extracted and Excel-extracted names go through fuzzy match + human confirmation.
9. **Public static web archive ships in v1** (revised by /autoplan CEO premise check). Baseball-Reference-style athlete profile + season leaderboard built as a static site (Astro or Next.js static export, decided at Milestone 8). Generated from canonical.jsonl, hosted free on GitHub Pages or Cloudflare Pages. The wireframe and design brief from /plan-design-review are the visual reference. This makes MNEMEX visible to the 95% of the audience (fans, federation officials browsing on mobile) who won't run the CLI themselves. Bigger v1 scope, but ships the magical moment to the actual audience instead of just to CLI-running developers.
10. **Export hub is v1, but only three formats in v1.** CSV, JSON, and the legacy 9-column XLSX cover federations and the most common analyst use cases. Parquet and SQLite exports are deferred to v1.1 — they're nice-to-haves nobody has explicitly asked for, and pandas can produce them ad-hoc until the user-volume justifies first-class support.
11. Repo and product rename to **MNEMEX** before any external visibility.
12. **MNEMEX canonical schema is broader than STRATHMARK's `HistoricalResult`.** Native support for pairs/teams (up to N competitors per row), multi-run events (array of run results + final-score policy), `score_type` enum (`time` / `hits` / `distance` / `raw_score`), DQ taxonomy as a controlled enum, knowledge events with no time/wood, school-team affiliation per event, eligibility status per athlete-year (college / pro / both).
13. **Source-aware parsers are real engineering work.** v1 ships five ingest paths: STIHL HTML scraper (refactor of existing), Vision-LLM Excel parser (handles arbitrary college host layouts), Vision-LLM PDF/newsletter parser (ALA, CANLOG, similar association publications — multi-page PDFs with mixed-layout result lists, show calendars, classifieds), Vision-LLM image parser (Arden Cogar-style photographed scoresheets), manual CSV paste-in.
14. **Athlete identity dedup has its own data store**, its own review queue, its own audit trail. Every merge/split is logged. Cross-source dedup is the load-bearing engineering problem of MNEMEX.
15. **STRATHMARK exports Tier 1 + Tier 2 only. Tier 3 stays in MNEMEX forever.** STRATHMARK is a handicap calculator. Tier 1 events (time-scored speed events including HORIZONTAL_SPEED and VERTICAL_SPEED for the college variants) are STRATHMARK 0.4.1-compatible and ship v1. Tier 2 events (HORIZONTAL_HARDHIT and VERTICAL_HARDHIT — hits-based) need STRATHMARK to grow a `score_type` field; v1 *implements but feature-flag-disables* the Tier 2 adapter. Tier 3 events (Caber Throw, Axe Throw, Speed Axe, Birling, Pulp Toss, Dendrology, Timber Cruise, Traverse, Steeple Chase, Wrapper Throw, Team Relay, Pole Climb, Choker Race) are captured in MNEMEX's canonical store and exposed via the export hub (CSV/JSON/Excel) for federation use, but they NEVER flow to STRATHMARK because handicap math doesn't model distance, hits, or knowledge tests. STRATHMARK's identity as a handicap engine is preserved.
16. **DQ / scratch / DNF use a controlled enum**, not free text. Per-source mapping tables normalize "dq-time" / "DQ time" / "DQ-T" → `DQ_TIME`.
17. **College wood metadata is NULL with provenance flag.** `wood_species=NULL, wood_diameter_mm=NULL, wood_quality=NULL, wood_metadata_source='not_published'`. Future work: capture wood specs from event programs separately and join.

## Approaches Considered

### Approach A — Pure static archive (rejected)

Static site generator from JSON, no backend. Excellent for the public archive surface, but image ingest, Vision-LLM Excel parsing, and human review queues all need server-side execution. Pure static doesn't fit v1.

### Approach B — Full backend + DB + API (deferred to v2/v3)

SQLite or Postgres canonical store, FastAPI backend, real-time data, full-text search via FTS5. Right architecture for scale, overkill for v1 given the audience size (federations + hardcore fans, probably <500 weekly users for a year).

### Approach C — Notebook / pip-package only (rejected)

Skipped entirely. Federations and most hardcore fans will not `pip install` anything. Wrong audience match.

### Approach A+ — Static Archive + Serverless Ingest (RECOMMENDED, locked in)

Hybrid. Browse layer is static (CDN-served HTML, free hosting). Ingest layer is serverless functions (Cloudflare Workers, Vercel Functions) for image upload + Vision LLM call + review queue. **Canonical store for v1 is JSONL files committed to a `data/` branch in the MNEMEX git repo.**

Why JSONL+git for v1:
- The backup story already mandates daily JSONL exports to git (see Operational Concerns). Making the same JSONL the *primary* store eliminates a sync layer.
- Git history is the audit trail.
- 100K rows fits comfortably; even 1M rows fits.
- No managed-service auth, no per-row egress cost, no migrations.
- Migration to Turso / D1 / Neon is a v1.1+ decision when concurrent write throughput becomes the bottleneck — not v1.

### Canonical schema

```python
# mnemex/schema.py

from dataclasses import dataclass, field
from datetime import date as Date
from enum import Enum
from typing import Optional

class ScoreType(str, Enum):
    TIME = "time"             # seconds; lower is better
    HITS = "hits"              # integer count; lower is better
    DISTANCE = "distance"      # meters / feet; higher is better (caber, axe-throw distance)
    RAW_SCORE = "raw_score"    # written-test points; higher is better
    PLACE_ONLY = "place_only"  # finishing position published, no time (e.g., axe throw,
                               # log roll in ALA newsletters). Place is the score.

class Division(str, Enum):
    OPEN = "open"
    MASTERS_INTERMEDIATE = "M/I"   # ALA / college Masters/Intermediate division
    NOVICE_INTERMEDIATE = "N/I"    # ALA Novice/Intermediate
    WOMEN = "women"
    MEN = "men"
    MIXED = "mixed"
    JUNIOR = "junior"
    PRO = "pro"
    ROOKIE = "rookie"

class DQReason(str, Enum):
    DQ_TIME = "dq_time"             # exceeded time limit
    DQ_DISC = "dq_disc"             # missed discriminator (e.g., specific cut on saw events)
    DQ_CUT_OUT = "dq_cut_out"       # saw event ended improperly
    DQ_FELL = "dq_fell"             # fell off platform
    DQ_FOOTHOLD = "dq_foothold"     # foothold violation
    DQ_SCOOPS = "dq_scoops"         # axe scooping violation
    SCRATCH = "scratch"             # withdrew before competing
    DNF = "dnf"                     # did not finish

class ExtractionStatus(str, Enum):
    """Status field on rows in the `pending` queue. Distinct from DQReason
    (which is a row-level competitive outcome) and from the review-queue
    state machine (which tracks pending → verified → committed)."""
    OK                = "ok"                  # extraction succeeded; row is fully populated
    FAILED_TRANSIENT  = "failed_transient"    # network/HTTP error after retry budget exhausted
    FAILED_TIMEOUT    = "failed_timeout"      # per-call timeout exceeded
    FAILED_PARSE      = "failed_parse"        # model returned non-JSON or schema-invalid JSON
    PARTIAL           = "partial"             # JSON valid but row count below source count
    PENDING_STALE     = "pending_stale"       # has been pending > 14 days; surfaced in review --failures
    PENDING_BUDGET    = "pending_budget"      # deferred to next budget window (cost ceiling hit)
    NEEDS_INPUT       = "needs_input"         # identity dedup requires external (human) name lookup;
                                              # used when first-name-only entries have no contextual hint

class Discipline(str, Enum):
    # Each enum value corresponds to exactly one real-world event. No overloading.
    # Standing Block Chop and Single Buck are distinct events and get distinct keys.
    UNDERHAND = "UNDERHAND"           # Underhand Chop (axe, horizontal block)
    STANDING_BLOCK = "STANDING_BLOCK" # Standing Block Chop (axe, vertical block)
    SINGLE_BUCK = "SINGLE_BUCK"       # one person, crosscut saw
    DOUBLE_BUCK = "DOUBLE_BUCK"       # pair, crosscut saw
    JACK_AND_JILL = "JACK_AND_JILL"   # mixed-gender pair, crosscut saw
    HORIZONTAL_HARDHIT = "HORIZONTAL_HARDHIT"   # HHH — hits to break horizontal block
    HORIZONTAL_SPEED  = "HORIZONTAL_SPEED"      # HS / HSC — time on horizontal block
    VERTICAL_HARDHIT  = "VERTICAL_HARDHIT"      # VH / VHH — hits to break vertical block
    VERTICAL_SPEED    = "VERTICAL_SPEED"        # VS / VSC — time on vertical block
    OBSTACLE_POLE     = "OBSTACLE_POLE"         # OP
    POLE_CLIMB        = "POLE_CLIMB"
    CHOKER_RACE       = "CHOKER_RACE"
    POWER_SAW         = "POWER_SAW"
    HOT_SAW           = "HOT_SAW"
    STOCK_SAW         = "STOCK_SAW"
    SPRINGBOARD_1BD   = "SPRINGBOARD_1BD"       # one-board springboard
    SPRINGBOARD_2BD   = "SPRINGBOARD_2BD"       # two-board springboard
    AXE_THROW         = "AXE_THROW"
    SPEED_AXE_THROW   = "SPEED_AXE_THROW"
    CABER_THROW       = "CABER_THROW"
    BIRLING           = "BIRLING"               # logrolling
    PULP_TOSS         = "PULP_TOSS"             # pair
    DENDROLOGY        = "DENDROLOGY"            # knowledge — tree ID
    TIMBER_CRUISE     = "TIMBER_CRUISE"         # knowledge — forestry estimation
    TRAVERSE          = "TRAVERSE"
    STEEPLE_CHASE     = "STEEPLE_CHASE"
    WRAPPER_THROW     = "WRAPPER_THROW"
    TEAM_RELAY        = "TEAM_RELAY"

    # Aliases for cross-source mapping. The ingest layer's `discipline_alias_map`
    # normalizes "SB" / "STANDING BLOCK" / "Standing Block Chop" → STANDING_BLOCK,
    # and "SB" (in college context) / "Single Buck" → SINGLE_BUCK based on
    # source_circuit context. STIHL-context "SB" = STANDING_BLOCK; college-context
    # "SB" = SINGLE_BUCK. The mapping is documented in
    # mnemex/ingest/discipline_aliases.py and is the single source of truth.

class FinalScorePolicy(str, Enum):
    SINGLE_RUN     = "single_run"        # one run, value is final
    BEST_OF_RUNS   = "best_of_runs"      # min (time/hits) or max (distance/raw_score)
    AVERAGE_OF_RUNS= "average_of_runs"   # arithmetic mean of completed runs
    SUM_OF_RUNS    = "sum_of_runs"       # rare; documented for forward-compat

class EventCircuit(str, Enum):
    STIHL_PRO       = "stihl_pro"
    AWFC_COLLEGE    = "awfc_college"
    ALA_SHOW        = "ala_show"
    CANLOG_SHOW     = "canlog_show"
    LWC             = "lwc"               # Lumberjack World Championship (Hayward, WI)
    REGIONAL        = "regional"
    UNKNOWN         = "unknown"           # forced fallback; review queue MUST resolve

@dataclass(frozen=True)
class CompetitorRef:
    canonical_id: str           # MNEMEX-issued canonical athlete ID
    name_as_recorded: str       # exact spelling from source (preserved for audit)
    school_team: Optional[str] = None    # e.g., "UM-A" — nullable for pro events
    division: Optional[str] = None       # "Open", "Master", "Junior", etc.

@dataclass(frozen=True)
class RunResult:

    # results
    runs: list[RunResult] = field(default_factory=list)

    final_score_policy: FinalScorePolicy = FinalScorePolicy.SINGLE_RUN
    dq_reason: Optional[DQReason] = None   # row-level DQ (when no individual run completed)
    place: Optional[int] = None
    points: Optional[int] = None           # college-style place-points

    # wood (only when published; NULL otherwise)
    wood_species: Optional[str] = None
    wood_diameter_mm: Optional[int] = None
    wood_quality: Optional[int] = None     # 1..10; STRATHMARK convention
    wood_quality_imputed: bool = False
    wood_metadata_source: str = ""         # "published" | "not_published" | "imputed_default"

    # extraction metadata (image / Excel / PDF ingest only)

    extraction_notes: Optional[str] = None
    extraction_status: ExtractionStatus = ExtractionStatus.OK  # see ExtractionStatus enum
    identity_resolution_required: bool = False  # set when names are first-name-only,
                                                # initials-only, or fuzzy-match below threshold

# Schema invariants (asserted in tests/test_schema.py and at row construction):
#   1. If `runs` is non-empty:
#        `final_score` MUST equal apply_policy(runs, final_score_policy)
#        where apply_policy is defined per FinalScorePolicy:
#          SINGLE_RUN       → runs[0].value (with the run-level dq_reason becoming row-level)
#          BEST_OF_RUNS     → min(values) for ScoreType.TIME / HITS;
#                             max(values) for ScoreType.DISTANCE / RAW_SCORE
#          AVERAGE_OF_RUNS  → arithmetic mean of completed (non-DQ) runs
#          SUM_OF_RUNS      → sum of completed runs
#   2. If `runs` is empty:
#        `final_score` is the source-published number; `final_score_policy=SINGLE_RUN`.
#   3. For ScoreType.RAW_SCORE (knowledge events: Dendrology, Timber Cruise):
#        `final_score` holds the raw integer-valued count cast to float (e.g., "17 of 20" → 17.0).
#        The maximum-possible value is recorded in `extraction_notes` (e.g., "max=20").
#        Future enrichment can compute a normalized 0..1 score; v1 stores the raw count only.
#   4. For ScoreType.PLACE_ONLY:
#        `final_score` = `place` cast to float (e.g., 1st → 1.0). `runs` is empty.
#        `place_only=true` events do not flow to STRATHMARK Tier 1/2/3 in v1
#        (no time/hits/distance available).
#   5. For row-level DQ (`dq_reason` set, no completed run):
#        `final_score=None`; STRATHMARK projection skips the row.

# Per-discipline default FinalScorePolicy. Ingest parsers use this when the source
# does not specify policy explicitly. Override at the row level when the source DOES
# specify (e.g., college sheets often show "Time 1, Time 2, Ave" — that's AVERAGE_OF_RUNS).
DEFAULT_SCORE_POLICY: dict[Discipline, FinalScorePolicy] = {
    Discipline.UNDERHAND:           FinalScorePolicy.SINGLE_RUN,
    Discipline.STANDING_BLOCK:      FinalScorePolicy.SINGLE_RUN,
    Discipline.SINGLE_BUCK:         FinalScorePolicy.SINGLE_RUN,

    Discipline.JACK_AND_JILL:       FinalScorePolicy.SINGLE_RUN,
    Discipline.OBSTACLE_POLE:       FinalScorePolicy.AVERAGE_OF_RUNS,  # 1st run + 2nd run
    Discipline.POLE_CLIMB:          FinalScorePolicy.BEST_OF_RUNS,     # 1st climb + 2nd climb
    Discipline.CHOKER_RACE:         FinalScorePolicy.BEST_OF_RUNS,     # 1st run + 2nd run
    Discipline.HORIZONTAL_SPEED:    FinalScorePolicy.AVERAGE_OF_RUNS,  # Time 1, Time 2, Ave
    Discipline.VERTICAL_SPEED:      FinalScorePolicy.AVERAGE_OF_RUNS,

    Discipline.VERTICAL_HARDHIT:    FinalScorePolicy.AVERAGE_OF_RUNS,
    # Tier 3: no STRATHMARK projection in v1, but policy is still recorded for archive
    Discipline.AXE_THROW:           FinalScorePolicy.SINGLE_RUN,       # PLACE_ONLY in ALA
    Discipline.BIRLING:             FinalScorePolicy.SINGLE_RUN,

TIER2_DISCIPLINES = {

# NOTE on time-scored axe events (HORIZONTAL_SPEED, VERTICAL_SPEED): these
# are time-scored and structurally compatible with STRATHMARK 0.4.1 today.
# They are CURRENTLY in Tier 1 — added to TIER1_DISCIPLINES below — and
# export cleanly with the existing handicap math. The earlier draft of this
# doc miscategorized them as Tier 2; corrected here.

TIER3_DISCIPLINES = {
    # Outside STRATHMARK's handicap-engine identity. Either ScoreType.DISTANCE,
    # ScoreType.RAW_SCORE, ScoreType.PLACE_ONLY, or pair/team aggregations
    # that don't fit the time-gap model. MNEMEX captures these in the
    # canonical store; STRATHMARK exports activate when STRATHMARK adds a
    # non-handicap scoring module (post-v1).
    Discipline.AXE_THROW,
    Discipline.SPEED_AXE_THROW,
    Discipline.CABER_THROW,
    Discipline.BIRLING,
    Discipline.PULP_TOSS,
    Discipline.DENDROLOGY,
    Discipline.TIMBER_CRUISE,
    Discipline.TRAVERSE,
    Discipline.STEEPLE_CHASE,
    Discipline.WRAPPER_THROW,
    Discipline.TEAM_RELAY,
    Discipline.POLE_CLIMB,    # uses pole equipment STRATHMARK has no model for
    Discipline.CHOKER_RACE,   # multi-run + obstacle structure outside handicap math
}

# Sanity invariant (asserted in tests/test_schema.py):
#   set(Discipline) == TIER1_DISCIPLINES | TIER2_DISCIPLINES | TIER3_DISCIPLINES
#   TIER1 ∩ TIER2 == TIER1 ∩ TIER3 == TIER2 ∩ TIER3 == ∅

# Mapping from MNEMEX Discipline → string token STRATHMARK expects in
# HistoricalResult.discipline. Each enum value maps to a UNIQUE token; we do
# NOT collapse Standing Block and Single Buck into the same "SB" token even
# though both events are colloquially called "SB" by their respective circuits.
# Collapsing them would corrupt downstream handicap math: STRATHMARK trains
# its predictor per discipline, and Standing Block (vertical-block axe chop)
# and Single Buck (one-person crosscut saw) have completely different
# kinematics. The tokens below are explicit and unambiguous.
STRATHMARK_DISCIPLINE_MAP: dict[Discipline, str] = {
    Discipline.UNDERHAND:        "UH",        # Underhand Chop
    Discipline.STANDING_BLOCK:   "STB",       # Standing Block Chop (NOT "SB" — see note above)
    Discipline.SINGLE_BUCK:      "SBUCK",     # Single Buck saw event
    Discipline.DOUBLE_BUCK:      "DBUCK",     # Double Buck (pair)
    Discipline.JACK_AND_JILL:    "JJ",        # Jack & Jill (mixed pair)
    Discipline.OBSTACLE_POLE:    "OP",
    Discipline.POWER_SAW:        "PSAW",
    Discipline.HOT_SAW:          "HSAW",
    Discipline.STOCK_SAW:        "SSAW",
    Discipline.SPRINGBOARD_1BD:  "SPR1",
    Discipline.SPRINGBOARD_2BD:  "SPR2",
}

# IMPORTANT: STRATHMARK 0.4.1's existing data uses tokens "SB" and "UH" only
# (its prior data source was the 9-column STIHL Excel format where "SB" meant
# Standing Block). The token migration is part of the coordinated MNEMEX v1
# integration: at Milestone 0, MNEMEX submits a STRATHMARK PR adding the
# expanded token vocabulary above to STRATHMARK's discipline registry. Until
# that PR lands and STRATHMARK 0.4.2 (or whatever version contains it)
# releases, MNEMEX exports use the legacy 2-token vocabulary AND restricts
# Tier 1 export to {UNDERHAND, STANDING_BLOCK} only. This is captured as a
# "Cross-repo coordination item" below.

def to_strathmark_results(rows: list[CanonicalRow]) -> list[HistoricalResult]:
    """
    Project Tier 1 MNEMEX rows down to STRATHMARK HistoricalResult records.

    Rules:
      - Tier 2 / Tier 3 disciplines are filtered out (logged via mnemex.logger).
      - Multi-run events (e.g., OBSTACLE_POLE with AVERAGE_OF_RUNS): row.final_score
        is already the canonicalized value per FinalScorePolicy (Schema invariant 1).
        STRATHMARK receives that scalar. We do NOT pass per-run values; STRATHMARK's
        v0.4.1 HistoricalResult is single-time. Future STRATHMARK extension may add
        per-run support; until then, the projection is intentionally lossy on runs.
      - Pair events (DOUBLE_BUCK, JACK_AND_JILL) emit one HistoricalResult per
        competitor with the same time. STRATHMARK treats them as individual results.
      - DQ rows are skipped (HistoricalResult requires a numeric time).
      - Quality defaults to 5 with `quality_imputed` flag preserved in MNEMEX store
        but not propagated (STRATHMARK 0.4.1 has no slot for it; future bump may add).
      - Discipline string mapped via STRATHMARK_DISCIPLINE_MAP (NOT row.discipline.value)
        to ensure STRATHMARK sees its expected vocabulary.
    """
    out = []
    for row in rows:
        if row.discipline not in TIER1_DISCIPLINES:
            mnemex.logger.debug("tier1_filter_skip", discipline=row.discipline,
                                reason="not_tier1", row_id=row.source_native_id)
            continue
        if row.dq_reason is not None or row.final_score is None:
            mnemex.logger.debug("tier1_filter_skip", discipline=row.discipline,
                                reason="dq_or_no_score", row_id=row.source_native_id)
            continue
        if row.score_type != ScoreType.TIME:
            mnemex.logger.debug("tier1_filter_skip", discipline=row.discipline,
                                reason="non_time_score_type",
                                row_id=row.source_native_id)
            continue

        species = row.wood_species or "Unknown"
        diameter = row.wood_diameter_mm or 0
        quality = row.wood_quality if row.wood_quality is not None else 5
        strathmark_discipline = STRATHMARK_DISCIPLINE_MAP[row.discipline]

        for competitor in row.competitors:
            out.append(HistoricalResult(
                discipline=strathmark_discipline,
                time=row.final_score,
                species=species,
                diameter_mm=diameter,
                quality=quality,
                date=row.event_date,
            ))
    return out
```

**Field-loss table.** The Tier 1 projection necessarily drops MNEMEX-specific fields that STRATHMARK's `HistoricalResult` does not model. This is intentional — MNEMEX retains them in its own canonical store. The contract test asserts that no Tier 1 row is silently *omitted* (only its richer fields are flattened):

| MNEMEX field          | STRATHMARK field   | Behaviour in projection                          |
|-----------------------|--------------------|--------------------------------------------------|
| `discipline`          | `discipline`       | Mapped via `STRATHMARK_DISCIPLINE_MAP`           |
| `final_score` (TIME)  | `time`             | Direct                                           |
| `wood_species`        | `species`          | Defaults to `"Unknown"` if NULL                  |
| `wood_diameter_mm`    | `diameter_mm`      | Defaults to `0` if NULL (STRATHMARK fallback)    |
| `wood_quality`        | `quality`          | Defaults to `5` if NULL (with `quality_imputed`) |
| `event_date`          | `date`             | Direct                                           |
| `competitors[i].name` | `CompetitorRecord.name` | One emitted result per competitor in the row |
| `division`, `gender`, `event_name`, `event_url`, `place`, `points`, `special_markers`, `extraction_confidence`, `extraction_notes`, `source`, `verified_at`, `verified_by` | (none)             | Dropped at projection (preserved in MNEMEX store) |

**Contract test scope.** `tests/test_strathmark_contract.py` validates three layers:

1. **Type shape.** `to_strathmark_results(fixtures)` returns `list[HistoricalResult]` with all fields populated.
2. **Round-trip.** The result list is serialized via `mnemex/strathmark_adapter/jsonl.py` to a JSONL file, then STRATHMARK's `import_legacy.py --dry-run --input <file>` is invoked as a subprocess; non-zero exit fails the test. This catches changes in STRATHMARK's import-validator that the type shape alone would miss.
3. **No-silent-loss.** For every fixture row where `discipline in TIER1_DISCIPLINES and dq_reason is None and score_type == TIME`, the projection emits ≥ 1 `HistoricalResult` (one per `competitor` in the row). If 0 are emitted, the test fails AND a structured log line records the filter reason for the dropped row.

Schema drift on either side fails CI. STRATHMARK upgrade protocol below specifies how to handle that failure deliberately.

### v1 integration: JSONL only

**v1 ships exactly one STRATHMARK integration path.** Everything else is designed but deferred.

```bash
mnemex export strathmark --format jsonl --output ./out/strathmark-tier1-YYYY-MM-DD.jsonl
mnemex export strathmark --format jsonl --output ./out/... --dry-run    # writes file, no validation invoked
```

STRATHMARK's `import_legacy.py` consumes the file with `--commit` after a `--dry-run` validation, identical to its existing legacy-Excel ingest path.

Properties:
- File-based; audit-trail in git history of the export.
- Asynchronous; no live coupling to a running STRATHMARK instance.
- Idempotent re-imports use a natural row key (`source + source_native_id + competitor.canonical_id + run_index`) — STRATHMARK's `import_legacy.py` already deduplicates on this key.

### Deferred to v1.1+: HTTP POST and Supabase direct write

**Why deferred.** They serve a single use case — live tournament ingest where MNEMEX-extracted rows need to flow into a running STRATHMARK instance within seconds. That scenario is STRATHEX-driven and post-v1. Building both paths now means: bearer-token auth, partial-failure retry, idempotency keys, integration tests against live and staging Supabase, dry-run plumbing through HTTP, and ~3-5 days of work that v1 simply doesn't need.

**Forward-compatibility.** The Tier 1 projection logic in `mnemex/strathmark_adapter/tier1.py` is path-agnostic — it produces a list of `HistoricalResult` objects. v1 has one serializer (`jsonl.py`); v1.1 will add `http.py` and `supabase.py` against the same projection function. No schema change is required to add them later.

**Stub files.** `mnemex/strathmark_adapter/http.py` and `supabase.py` exist as empty modules with `NotImplementedError("v1.1+")` so the package layout is stable.

### Identity dedup service

```python
# mnemex/identity.py

@dataclass
class CanonicalAthlete:
    canonical_id: str           # MNEMEX-issued, stable across renames
    primary_name: str           # canonical display name
    aliases: list[str]          # all observed name variants
    stihl_athlete_id: Optional[str] = None    # /Athlete/{id} when known — gold seed
    school_team_history: list[tuple[int, str]] = field(default_factory=list)  # [(year, team)]
    eligibility: list[tuple[int, str]] = field(default_factory=list)  # [(year, "college"|"pro"|"both")]
    notes: Optional[str] = None
```

**Matching rules:**
- STIHL athlete URLs are gold — same URL = same canonical athlete, **no review needed for the identity link** (the data row still goes through normal review for time/score correctness).
- Image / Excel-extracted / newsletter-extracted names go through:
  1. Exact-match on canonical name AND last-name present (not first-name-only) → identity *proposed* with `match_confidence=1.0`. The proposal is staged in `pending`; reviewer one-click confirms.
  2. Fuzzy match (token sort ratio + last-name + first-initial) ≥ 95% AND last-name present → identity *proposed* with `match_confidence={fuzzy_score}`. Same staging, one-click confirm.
  3. First-name-only or initials-only entries → ALWAYS enter identity-review queue with no automatic proposal, regardless of any fuzzy score against existing canonical athletes (per Feasibility 3 in the prior review).
  4. Below 95% fuzzy or ambiguous (multiple candidates within 2% of each other) → identity-review queue with the top 3 candidates surfaced to the reviewer.
- **No identity match auto-commits to the canonical store.** The constraint "no unverified row reaches the handicap engine" requires every identity link to pass through human review, even at fuzzy score 1.0. The 95% threshold determines whether the row enters review pre-populated with a proposal (one-click) or unpopulated (manual lookup) — not whether it bypasses review.
- Every link / merge / split is logged to an append-only audit log. Reversible (see "Identity merge and split — propagation to STRATHMARK" in Operational Concerns).

**Module ownership (load-bearing — disambiguates split responsibility):**

```
mnemex/identity.py   OWNS: matching algorithm, canonical store, audit log
                     EXPOSES: match(name, context) -> Match | NeedsReview
                              commit_link(canonical_id, name, source) -> None
                              split(canonical_id, row_assignments) -> (id_a, id_b)

mnemex/review.py     OWNS: review-queue state machine, CLI UI, lock semantics
                     CALLS: identity.match() during ingest commit
                            identity.commit_link() when a reviewer approves a NeedsReview
                            identity.split() when a reviewer reverses a wrong merge
                     STORES: pending / verified / committed rows
                     NEVER: contains matching logic itself
```

A row entering `pending` carries `match_result: Match | NeedsReview`. If `Match`, the reviewer sees the proposed `canonical_id` and can approve it (→ `commit_link`) or reject it (→ `NeedsReview`). If `NeedsReview`, the reviewer either picks an existing canonical athlete from a fuzzy-search picker or creates a new one. All paths record an audit-log entry.

### Review queue states

```
pending  →  verified  →  committed
   |          |              |
   ↓          ↓              ↓
rejected   rejected     immutable (only revertable via audit log entry)
```

- **Pending:** extraction returned data, awaiting human review.
- **Verified:** human approved the row + the athlete identity match. NOT yet visible to exports.
- **Committed:** integrated into the canonical store. Visible to STRATHMARK adapter and export hub.
- **Rejected:** explicitly marked bad. Stays in the queue for audit; never reaches the store.

The committed → reverted path requires an audit-log entry (source-level recall, e.g., "host coach corrected the original Excel; reingest"). This preserves STRATHMARK reproducibility — a handicap mark generated yesterday must be reproducible today.

**Storage layout (file paths and shapes — load-bearing, do not deviate):**

```
data/
├── canonical.jsonl           # COMMITTED rows only. One CanonicalRow per line.
│                             # The export hub and STRATHMARK adapter read from here.
│                             # Append-only. Tracked in git (audit trail).
├── pending.db                # SQLite database. PENDING + VERIFIED rows + extraction
│                             # metadata. Mutable working state. NOT tracked in git.
│                             # Schema: rows(id PK, review_state, extraction_status,
│                             # row_json, created_at, updated_at, locked_by, locked_at).
│                             # Indexes on review_state, extraction_status, created_at.
├── rejected.jsonl            # REJECTED rows. Append-only audit. Tracked in git.
├── identity_canonical.jsonl  # The CanonicalAthlete records. Append-only. Git-tracked.
├── identity_audit.jsonl      # Identity link / merge / split events. Append-only. Git-tracked.
├── cost_ledger.jsonl         # Vision-LLM call cost records. Append-only. Git-tracked.
├── corrections/              # Generated correction files for STRATHMARK propagation.
│   ├── merge-YYYY-MM-DD.jsonl
│   └── split-YYYY-MM-DD.jsonl
└── .review.lock              # PID-file for single-reviewer constraint.

.gitignore additions:
  data/pending.db
  data/pending.db-journal
  data/.review.lock
```

**Storage architecture rationale.** Two different tools for two different jobs:
- **Canonical store (JSONL+git):** immutable historical record. Append-only. Audit trail = git history. Re-readable from scratch for every export. Solves the backfill-scale failure mode for the *committed* corpus by being write-once-read-many.
- **Pending queue (SQLite):** mutable working state. Indexed lookups by review_state during `mnemex review`. Transactions on state transitions. Survives backfill scale (10K-100K pending rows is trivial for SQLite). Per-row optimistic locking via `locked_by/locked_at` columns. NOT in git — it's working state, not history.

State transitions:
- `pending` → `verified`: SQLite UPDATE on the row; reviewer's identity recorded.
- `verified` → `committed`: SQLite SELECT + transaction → append to `canonical.jsonl` → SQLite DELETE. Atomic via SQLite transaction wrapping the file append (fsync before commit).
- any → `rejected`: SQLite SELECT → append to `rejected.jsonl` → SQLite DELETE. Same atomicity.

All writes to JSONL files go through `mnemex/store.py` which uses `fcntl.flock` on POSIX and `msvcrt.locking` on Windows for atomicity. SQLite handles its own locking (WAL mode, default).

Append-only JSONL files can be tail-read by external tools without locking. The exports hub re-reads `canonical.jsonl` from scratch for every export — small files, predictable behaviour, no caching layer to invalidate.

**Backfill mode.** When `mnemex backfill --source <path> --batch-size 100` is invoked, ingest runs in batches: extract 100 rows → write to SQLite pending → exit. The user runs `mnemex review` to drain. Repeat. This bounds the human-reviewer queue depth and avoids the "10,000 row backlog of doom" failure mode. The batch size is configurable; 100 is a sane default for solo use.

### Tier 3 in STRATHMARK — open architectural question

The user's choice is "Tier 1 + 2 + 3 — STRATHMARK becomes the universal Timbersports engine."

This is a significant identity shift for STRATHMARK. Today it is a *handicap engine*: it predicts time gaps so faster athletes give slower athletes a head start. Tier 3 events (Caber Throw, Dendrology, Birling, Team Relay) don't have a time-gap structure — caber is distance, dendrology is a written test, team relay is aggregate, birling is duration-of-stay. There is no meaningful "handicap mark" for a written test.

Two viable paths inside STRATHMARK:

1. **Universal scoring engine.** STRATHMARK grows a `ScoreType` field and a per-score-type prediction strategy. Handicap math becomes one strategy among many. Substantial refactor of `predictor.py`, `variance.py`, `calculator.py`. Multi-release effort.
2. **Sibling module under STRATHMARK.** New module `strathmark.scoreboard` (or similar) that captures Tier 3 events as pass-through archive. Handicap engine stays focused on what it does well; non-handicap events are archived but don't go through the prediction cascade.

**MNEMEX captures Tier 3 from v1 either way.** The data is collected, stored, and exportable to CSV/JSON/Parquet for analysts and federations. The STRATHMARK Tier 3 export adapter is *implemented but disabled* in v1, ready to flip on whenever STRATHMARK lands the corresponding feature.

This decision belongs in STRATHMARK's roadmap, not on MNEMEX's v1 critical path. Flag it as an open item.

## CLI Surface Design

(Added by /plan-design-review.) v1 is a CLI tool. The CLI IS the user-facing experience for v1; "design" here means what the user sees on their screen, not visual layout.

### Voice and ethos (inherited from STRATHMARK)

- Plain text output. No emojis. No ANSI color codes. No box-drawing characters that don't render in basic terminals.
- Utility language. Status, action, orientation. Not mood, brand, aspiration.
- One thing at a time. Ingest commands print one line per row processed. Review commands clear and redraw between rows.
- Error messages name the file/path/value involved and propose the next action. "Couldn't read this image" is wrong; "Could not parse Vision-LLM response for image:abc123 — see extraction_notes; rerun with `mnemex review --failures` to retry" is right.
- No happy talk. No "Welcome to MNEMEX!" banners. No celebration of progress. The user knows what they're doing; the tool gets out of the way.

**Picker text-filter behaviour:** the `[t]ype to filter` mode runs prefix-match first (fast, predictable, no surprises). If the typed string yields zero candidates after 3+ characters, the picker silently falls back to fuzzy match (typo-tolerant). The transition is invisible to the user — they just keep typing until results appear. This gives prefix's predictability for the common case and fuzzy's resilience for the typo case. Implementation: `rapidfuzz.process.extract` for the fuzzy fallback.

### Structured errors (added by /plan-devex-review)

When MNEMEX is invoked from a programmatic context (`mnemex --json-errors` flag, or library use), errors are emitted as structured JSON for downstream consumers:

```json
{
  "error": {
    "code": "RENDER_TOOL_MISSING",
    "message": "libreoffice not found on PATH",
    "platform": "win32",
    "fix": "Install LibreOffice from https://www.libreoffice.org/download/download/",
    "fix_command": "winget install LibreOffice.LibreOffice",
    "doc_url": "https://github.com/SquirmyWormy275/mnemex/blob/main/docs/errors/render-tool-missing.md"
  }
}
```

Every error class has:
- `code` — stable identifier for programmatic handling
- `message` — human-readable
- platform/context fields when relevant
- `fix` — text the user reads
- `fix_command` — command they run (when one exists)
- `doc_url` — link to a per-error docs page (Tier 2 from the DX Hall of Fame)

The complete error registry lives in `mnemex/errors.py` as a frozen dataclass-per-error. Tests assert that every `code` has a matching `docs/errors/<code>.md` file.

**Default behaviour for imputed rows:** **permissive** (rows export, warning prints). Rationale: most college and ALA data won't have wood metadata, and refusing them by default would mean the user can't export anything from those sources. The warning is informational; the data still flows. The user can pass `--strict` to refuse rows with imputed quality when they want a high-confidence subset (e.g., for handicap calibration runs).

### `mnemex review` row count and batching (added by /plan-devex-review)

The review session presents ALL pending rows by default. Use `--limit N` to cap the session. The counter `[N/M]` always shows position. After every committed row, the counter increments. On `[q]uit`, the session exits cleanly; resuming picks up at the last unreviewed row.

### Row-comment audit threads (added by /plan-devex-review, EXPANSION mode)

Reviewers can attach threaded comments to any row at any state (pending, verified, committed). Comments persist for the lifetime of the row and survive identity merge/split. They are NOT redacted unless the row is redacted; on redaction, comments are tombstoned the same way.

**Schema:**

```python
@dataclass(frozen=True)
class RowComment:
    comment_id: str             # UUIDv7 — sortable, monotonic
    row_canonical_id: str       # references the canonical row's stable identity key
    author: str                 # $USER on local machine; auth identity in multi-reviewer v2
    text: str                   # free-form plain text; no markdown rendering
    created_at: str             # ISO 8601 UTC
    parent_comment_id: Optional[str] = None  # threading via parent reference
    redacted: bool = False      # set when the parent row is redacted
```

**Storage:** `data/row_comments.jsonl` — append-only, git-tracked. One JSON object per line. Index built in memory at read time, keyed by `row_canonical_id`. At v1 scale (estimated <1K comments/year), in-memory indexing is fine; v1.1 may add SQLite indexing if needed.

**CLI surface during review:**

The user is a hardcore Timbersports fan or a federation admin running this on their own laptop. They've installed the package via pip, never used MNEMEX before. Here's what they should feel and what the tool gives them at each step.

### Emotional arc — the failure case

When extraction goes badly (LLM rate limit, malformed JSON, network drop), what does the user feel?

- **Dread** (something is wrong): NEVER. The tool detects errors, names them, and gives the next action. The user feels *informed*, not stuck.
- **Frustration** (I have to start over): NEVER. Partial progress is always saved to pending. Resume with `--resume`.
- **Doubt** (did I lose data?): NEVER. The audit log + git history makes recovery obvious. `mnemex doctor` is the explicit "everything OK?" check.

### AI-slop boundary (CLI implementation discipline)

The plan inherits STRATHMARK's plain-text ethos. Specific implementation rules to preserve it:

- **No emoji in any output.** Not in progress, not in success, not in errors. The single `✓` glyph used in the surface specs above is intentional and minimal; no other glyphs.
- **No ANSI color.** No `colorama`, no `rich.print`, no `click.secho(fg=...)`. If a user pipes output to a file, the file should be readable.
- **No spinner libraries with default themes.** If progress is needed, print one line per N events with a flush. No twirling unicode.
- **No "rich" library default styles.** If `rich` is used at all, it's for plain-text tables (`Table(box=None)`) and that's it.
- **No banner or ASCII-art logo on `mnemex` invocation.** The version is in `mnemex --version` if asked.
- **No marketing voice in help text.** "scrape the STIHL official site" not "✨ effortlessly extract pristine data from STIHL's database ✨".
- **No emoji or icons in commit messages, error notes, or audit log entries.**

These rules are enforced by a CI check at Milestone 7: `tests/test_cli_output_style.py` runs every CLI command against a smoke fixture and asserts the output matches a regex denylist (no emoji codepoints, no ANSI escape sequences, no banner art).

### Accessibility (v1 CLI)

CLI a11y for MNEMEX is constrained but real:

- **Terminal width portability.** All output renders correctly at 80 columns (the lowest-common-denominator terminal width). Tables wider than 80 cols use a "vertical mode" fallback (each row's fields stack on multiple lines). Detected via `shutil.get_terminal_size()`; 80 cols is the floor.
- **No reliance on color or styling.** Already enforced by the no-ANSI rule. A user with a monochrome terminal, a screen reader piping through `say`, or output redirected to a file gets identical information.
- **Keyboard-only review flow.** No mouse, no menus, no fancy line editor. Single-keystroke actions (`a`/`r`/`e`/`n`/`s`/`q`/`?`/digits). The `[t]ype to filter` mode in the identity picker uses the user's existing readline (arrows, ctrl-A/E, etc.).
- **Help discoverable inline.** `?` from any review prompt prints the keyboard reference. The user never has to leave the session to learn the keys.
- **Stable output ordering.** Same input → same output across runs (modulo timestamps). Important for automated pipelines and screen-reader users who navigate by recall.
- **No flashing, no rapid repaints.** The review screen redraws on action only; no auto-update timers.
- **Screen-reader friendly.** Each row's fields are labeled (`athlete:`, `time:`) — a screen reader doesn't need spatial alignment to follow the data. The PID-file lock message is one full sentence (`error: review session already in progress · held by PID ...`), readable end-to-end.

### Vision-LLM error handling and retry

Every Vision-LLM call (Excel parser, newsletter parser, image parser) MUST handle:

- **HTTP / network errors:** exponential backoff with jitter, base delay 1s, factor 2, max 5 attempts, max total wait 60s. Beyond max attempts → write extraction record to `pending` queue with `extraction_status=failed_transient` and surface in CLI `mnemex review --failures`.
- **Rate-limit errors (429):** respect `Retry-After` header if present; otherwise fall back to exponential backoff. Do NOT count rate limits against the 5-attempt budget.
- **Timeouts:** per-call timeout 60s for image, 120s for multi-page PDF. On timeout, fail to `pending` with `extraction_status=failed_timeout`.
- **Malformed JSON output:** model returned non-JSON or schema-invalid JSON. First retry with the same prompt + 1 reminder turn ("the previous response was not valid JSON; respond with only the JSON object matching the schema"). If still malformed, write the raw response to `pending` with `extraction_status=failed_parse` and the raw text in `extraction_notes` for human review.
- **Partial extractions:** model returned valid JSON but with fewer rows than the source contains (detected by row-count cross-check against a deterministic page-row counter). Write to `pending` with `extraction_status=partial`; reviewer can choose to accept partial or trigger reprocessing.
- **Stuck pending rows:** rows that have been `pending` for > 14 days get auto-flagged in the next CLI invocation as `pending_stale`. They are NOT auto-rejected. Stale rows count against the "review queue load" metric in the operational dashboard.

`extraction_status` is a distinct enum from `dq_reason` and lives on the pending row, not the canonical row. Once a row is `committed`, `extraction_status` is no longer relevant.

### Cost monitoring and ceilings

Vision-LLM inference is the only meaningful runtime cost. Pricing baseline (Claude Vision, May 2026): roughly USD 0.01 per image-page processed at 1024×1568 input resolution. At v1 volume estimates:

- Single ALA newsletter (6 pages) → ~USD 0.06 per newsletter.
- Single college Excel host file (40 sheets, rendered as images) → ~USD 0.40 per file.
- Single Arden image → ~USD 0.01 per image.
- Arden's archive (~5,000 images, hypothetical full backfill) → ~USD 50 one-time.
- Steady state (3 newsletters/month, 5 Excel files/season, 50 images/month) → ~USD 5/month.

Implementation:

- A `cost_ledger` JSONL file records every Vision-LLM call with `(timestamp, source, pages, estimated_cost_usd, request_id)`. The file lives at `data/cost_ledger.jsonl`, one line per call, append-only.
- **Default ceilings:** Daily USD 20, Monthly USD 200. Configurable via `MNEMEX_COST_DAILY_CEILING` and `MNEMEX_COST_MONTHLY_CEILING`. The previous USD 5/50 figures were too tight and would trip during normal Milestone 3 calibration runs.
- **Atomic check-and-reserve.** Before any Vision-LLM call, `cost_ledger.reserve(estimated_cost_usd)` does an atomic compare-and-swap: read the current daily total, compute proposed total, and *atomically* either (a) append a `RESERVED` row with the estimated cost and return a reservation token, or (b) raise `BudgetExceeded`. The atomicity primitive is a `fcntl.flock` exclusive lock on the ledger file for the duration of the read-write cycle (rare contention; v1 is single-writer, but the lock guards against accidental concurrent invocations).
- After the call completes, `cost_ledger.finalize(reservation_token, actual_cost_usd)` updates the row from `RESERVED` to `COMMITTED`. If the call failed mid-flight, `cost_ledger.cancel(reservation_token)` removes the reservation.
- **Kill-switch.** When `reserve()` raises `BudgetExceeded`, the calling code writes the extraction request to the `pending_budget` queue with `extraction_status=PENDING_BUDGET`. CLI `mnemex review --budget-deferred` lists these.
- **Pending-budget drain.** The next-day drain happens lazily: on every CLI/serverless invocation, if the day has rolled over and the new daily budget is healthy, queued `pending_budget` items are processed FIFO until the new ceiling is hit again. Processing a deferred row counts against the day it actually runs, not the day it was queued.
- **Provider-side rate limits (RPM/TPM).** Independent of cost. `mnemex/ingest/llm_client.py` enforces a configurable per-provider concurrent-request cap (default 4) and a requests-per-minute throttle (default 60 RPM, configurable via `MNEMEX_LLM_RPM`). Both apply across all ingest paths sharing the client. Exceeding the throttle blocks the caller (with a max-wait timeout); doesn't fail.
- **Cost figures above are baseline estimates and will be re-verified during Milestone 0** once the actual provider is selected (see Open Question 4). Numbers above are placeholders; the *ceilings* are deliberately set high enough to absorb the verification work itself.

### Concurrency contract — ingest vs review

**Ingest commands** (`mnemex stihl`, `mnemex college`, `mnemex newsletter`, `mnemex image`, `mnemex manual`) write directly to `data/pending.db` via SQLite write transactions. They do NOT take the review PID-file lock. Multiple ingest commands CAN run concurrently — SQLite WAL mode handles the writer serialization. New rows appear in the pending DB and will be picked up by the next reviewer query.

**Review command** (`mnemex review`) takes the PID-file lock AND uses SQLite per-row optimistic locking. The PID-file enforces *single-reviewer-at-a-time*; the per-row lock guards against the rare case where ingest and review interleave on the same row (shouldn't happen because ingest never updates pending rows once written, but defense-in-depth is cheap). A reviewer picks up a row via `UPDATE pending SET locked_by=$pid, locked_at=now() WHERE id=$id AND locked_by IS NULL RETURNING *`; releases via `UPDATE pending SET locked_by=NULL` on state-transition.

### Review-queue PID-file lock

The CLI `mnemex review` writes a PID-file lock at `data/.review.lock` containing `{"pid": 12345, "started_at": "2026-05-04T13:00:00Z", "host": "alex-laptop"}`. A second `mnemex review` invocation reads the file:
- If the PID is alive on the same host → fail with `MNEMEX_REVIEW_LOCKED, held by PID 12345 since <ts>`.
- If the PID is dead (process gone) or the host differs → log a stale-lock warning, take ownership, and proceed.
- `mnemex review --force-unlock` removes the lock unconditionally and writes a `force_unlock` entry to the audit log (with operator identity from `$USER`).
- Lock is released cleanly on normal exit (atexit handler) and on SIGTERM/SIGINT.

Multi-reviewer concurrency is post-v1 and would require optimistic concurrency control on the row state field with a `version` integer, per-row TTL locking, and a reviewer identity field on every state transition. Out of v1.

### Athlete identity redaction (privacy)

Real people are in the canonical store. If an athlete requests their data be removed (GDPR-equivalent right-to-erasure):

- `mnemex identity redact --canonical-id <id> --reason "<text>"` tombstones the canonical-athlete record: drop `primary_name`, `aliases`, `notes`; preserve a salted SHA-256 hash of the original primary name for audit reconstruction; mark `redacted=true`, `redacted_at`, `redacted_reason`.
- All `CanonicalRow` records referencing the canonical_id keep the row (the *result* is public-record sport data) but replace `competitor.name_as_recorded` with `[redacted]`. The numeric/event data stays.
- Redaction propagates to STRATHMARK via a `redaction.jsonl` correction file consumed by `import_legacy.py --redact`.
- A redacted canonical_id is never reused.
- Audit-log entry is permanent.

Redaction is rare and operator-driven. v1 ships the primitive; no automation.

### Backup and restore

The canonical store IS JSONL+git (decided in Approach A+ above). Backups are therefore native: every commit to the `data/` branch is a backup checkpoint. Additionally, a daily GitHub Action snapshots the `data/` branch HEAD into `backups/YYYY-MM-DD` to give a consistent named restore point regardless of intra-day commit churn.

- **Backup ordering (consistency).** A single backup snapshot includes both `data/canonical.jsonl` (rows) and `data/identity_audit.jsonl` (identity-service log). The backup writes them in this order: identity audit first, then canonical rows. Restore reads in the same order. This guarantees that any `canonical_id` referenced by a row was already known to the identity service at backup time. The restore-time consistency check (`mnemex restore --verify`) asserts every `competitor.canonical_id` exists in the identity store.
- **RPO (recovery point objective):** 24 hours for named snapshots; effectively zero for git HEAD (every commit is recoverable).
- **RTO (recovery time objective):** 1 hour. Restore process is `git checkout data && mnemex restore --verify` (loads the canonical store and runs the consistency check).
- The `backups/` branch is push-only; no force-pushes (enforced by branch protection rule).
- A monthly restore drill (`mnemex restore --dry-run --from latest`) verifies the backup is loadable. v1 runs this manually; a CI job is v1.1.

### Identity merge and split — propagation to STRATHMARK

Both merge and split corrections need to propagate to STRATHMARK so previously-exported rows get re-keyed.

**Merge** (discover Athlete A and Athlete B are the same person):

1. Reviewer invokes `mnemex identity merge --keep <id_a> --absorb <id_b>`.
2. All canonical rows referencing `id_b` are rewritten to reference `id_a`. `id_b` is tombstoned (kept in the identity store with `merged_into=id_a`, never reused).
3. A `merge` record is written to the identity audit log: `{old_id_b, kept_id_a, affected_row_count, ts, operator}`.
4. **STRATHMARK propagation:** if any row referencing `id_b` had been previously exported to STRATHMARK, MNEMEX writes `corrections/merge-YYYY-MM-DD.jsonl` containing `{"old_canonical_id": "<id_b>", "new_canonical_id": "<id_a>"}` mappings, plus the affected `HistoricalResult` rows under their new keys. The CLI emits a warning instructing the operator to apply via `strathmark import_legacy.py --merge-corrections corrections/merge-YYYY-MM-DD.jsonl`. STRATHMARK's `--merge-corrections` flag (post-v0.4.1) consumes the mapping and re-keys its own store.

**Split** (discover a previous merge was wrong):

1. Reviewer invokes `mnemex identity split --canonical-id <id> --rows-to-split-out <row_ids>`.
2. The original `canonical_id` is retained for one of the two athletes. The other gets a fresh `canonical_id`. The named rows are reassigned in-place.
3. A `split` record is written to the identity audit log: `{original_id, retained_id, new_id, affected_row_count, ts, operator}`.
4. **STRATHMARK propagation:** MNEMEX writes `corrections/split-YYYY-MM-DD.jsonl` containing the row-to-ID mapping for the moved rows, plus their `HistoricalResult` projections under the new ID. CLI warning instructs the operator to apply via `strathmark import_legacy.py --split-corrections corrections/split-YYYY-MM-DD.jsonl`.

For both flows: STRATHEX / Missoula-Pro-Am / downstream tooling that read from STRATHMARK pick up corrections on their next sync. Handicap marks calculated against the wrong identity become provably re-derivable from corrected data.

The merge-and-split paths are rare and operator-driven. v1 does not automate them; it just provides safe primitives. STRATHMARK's `--merge-corrections` and `--split-corrections` flags are STRATHMARK-side work (a coordinated v0.4.2 release in service of MNEMEX) — flagged in the STRATHMARK upgrade protocol.

### Review-queue throughput estimate

The review queue is the human-throughput bottleneck of MNEMEX, so v1 must size for a sustainable rate.

Steady-state load (estimated):
- **STIHL ingest:** ~50 events/year, ~10 results/event with stable URLs → 500 rows/year, of which ~ 5% (~25 rows) hit identity review (mostly initial seeding; once canonical IDs are linked, future STIHL rows auto-resolve).
- **College Excel:** ~20 host events/year (AWFC, regional college conclaves), ~150 results/event → 3,000 rows/year, of which ~30% (~900 rows) hit identity review (no stable IDs in college sheets; first-name-only entries common).
- **ALA / CANLOG newsletters:** ~12 newsletters/year, ~120 results/newsletter → 1,440 rows/year, of which ~40% (~575 rows) hit identity review (heavy first-name-only entries observed in Sept 2025 fixture).
- **Arden Cogar images:** ~50 images/year (steady-state, after backfill) → ~50 rows/year, ~80% hit identity review.

**Total steady-state:** ~5,000 rows/year, ~1,500 review-queue items/year, or **~30 review items per week**.

At 30 sec/row reviewer throughput (the ALA newsletter case study), that's ~15 minutes/week of review work. At 90 sec/row (worst case, image extraction with ambiguous athlete names), that's ~45 minutes/week. Both are sustainable for a single hobbyist contributor.

**Backfill load** (one-time, not steady-state): the ~5,000 Arden images and any historical ALA archives could push the initial review queue to several thousand rows. Backfill is not gated by v1 — it can be done incrementally over months without blocking the steady-state pipeline.

The CLI review UI must therefore support: keyboard-driven approval/rejection, batch-merge for confident athlete-identity matches, and a "review later" defer state. These are noted as v1 CLI requirements, baked into Milestone 5.

### Schema migration policy

MNEMEX's canonical schema will evolve. The rules:

- **Enums are append-only.** New enum values may be added at any time. Existing values are never renamed or removed in a minor release. (If absolutely required, deprecate first with a `_DEPRECATED` suffix, ship a release with both, then remove in the next major.)
- **New fields are optional.** Adding a new field to `CanonicalRow` requires a default value. Existing rows in the store remain valid.
- **Field renames or removals require a migration script and a major version bump** (e.g., MNEMEX 1.x → 2.x). The migration script is committed alongside the schema change and runs automatically on `mnemex migrate`.
- **STRATHMARK contract changes** trigger a coordinated release (see STRATHMARK upgrade protocol below).

The rationale: MNEMEX's store is the long-lived historical record. Schema instability corrupts handicap reproducibility, which corrupts STRATHMARK's correctness guarantees. Append-only is the price of correctness.

### STRATHMARK upgrade protocol

The `pyproject.toml` runtime range is `strathmark>=0.4.1,<0.6`. The contract-test CI job pins to an **exact** version (`strathmark==0.4.1`) so that unrelated STRATHMARK patch releases don't surface as MNEMEX CI failures.

When a new STRATHMARK release ships:

1. **Open a feature branch in MNEMEX**: `chore/strathmark-X.Y-bump`.
2. **Bump the contract-test pin** in `tests/test_strathmark_contract.py` (or the corresponding CI workflow) to `strathmark==X.Y.0`. Leave the runtime range alone unless the new version is outside it.
3. **Run the contract test locally** with the new pin. It will either pass (STRATHMARK release was internally compatible) or fail (schema/behaviour change).
4. **If pass:** open a PR, merge, tag a MNEMEX release.
5. **If fail:** read STRATHMARK's CHANGELOG, identify the breaking change, update `mnemex/strathmark_adapter/tier{1,2,3}.py` to accommodate the new shape, re-run the contract test, then PR.
6. **Special case — STRATHMARK 0.5 with `score_type` field lands:** flip the Tier 2 feature flag from disabled to enabled in the same MNEMEX release. Bump the contract-test pin to `==0.5.0`. The runtime range `<0.6` already permits adoption — no major bump required. Tier 2 disciplines (`HORIZONTAL_HARDHIT`, `VERTICAL_HARDHIT`, `HORIZONTAL_SPEED`, `VERTICAL_SPEED`) start exporting cleanly. Release notes call this out explicitly so STRATHEX / Missoula-Pro-Am operators know new data is flowing.
7. **Tier 3 is never exported to STRATHMARK.** STRATHMARK is a handicap calculator and stays one. Tier 3 events (Caber Throw, Axe Throw, Speed Axe, Caber, Birling, Pulp Toss, Dendrology, Timber Cruise, Traverse, Steeple Chase, Wrapper Throw, Team Relay, Pole Climb, Choker Race) live in MNEMEX's canonical store forever and are queryable via the export hub (CSV / JSON / Excel) for federation use, but they do not flow into STRATHMARK because handicap math doesn't apply to them. The previous draft of this doc implied STRATHMARK would eventually grow to export Tier 3; that was wrong. STRATHMARK's identity does not change.

The contract test is the load-bearing artifact. Without it, STRATHMARK's natural evolution would silently break MNEMEX's exports — exactly the failure mode this design exists to prevent.

## Data Rights and Governance (added by /autoplan Codex CEO challenge)

MNEMEX republishes data from federations, association newsletters, photographed scoresheets, and other sources. The data-rights surface is broader than just STIHL; it spans every ingest path. v1 ships a `docs/data-rights.md` page covering:

### Attribution policy

Every canonical row carries a `source` field (already in schema) of form `<source_type>:<instance_id>`. The web archive renders this prominently on every row, every athlete profile, every leaderboard cell. Examples on the rendered athlete page:
- "16.78s · 2024 STIHL World Championship · source: data.stihl-timbersports.com/Event/12345"
- "50.42s · 2025 ALA Philomath Frolic · source: ALA September 2025 newsletter, page 4"
- "29.56s · 2024 ALA Longview Go 4th · source: photographed scoresheet contributed by Arden Cogar Jr."

Source links open the original on a click. No row is ever shown without provenance.

### Takedown / redaction policy

Two scopes:
- **Athlete redaction** — covered by the existing `mnemex identity redact` primitive. An athlete requests removal of their own results; canonical rows are tombstoned per the Operational Concerns redaction protocol.
- **Federation takedown** — new in v1. A federation can request that all data sourced from their publications be removed from MNEMEX. CLI: `mnemex takedown --federation <slug> --reason "<text>"`. Drops all rows where `source` starts with the federation prefix. Audit-logged. Reversible by the operator with explicit re-ingest, never by accident.

### Terms-of-service stance

Per source, v1 documents:
- **STIHL website** — scraping at 1s/req delay (existing). robots.txt review pending at M0; if disallowed, the STIHL ingest path is moved to a "manual download → process locally" flow rather than active scraping.
- **ALA / CANLOG newsletters** — these are public PDFs distributed to members. v1 stance: republishing tabular results with source attribution is fair use under the editorial-summary doctrine. Not legally bulletproof, but defensible. The takedown primitive above is the safety valve.
- **AWFC college Excel scorebooks** — federation backfill via `mnemex publish` only. The federation explicitly opts in by uploading; consent is implicit in the upload action.
- **Photographed scoresheets** — v1 only ingests images the operator (you) is licensed to use (your own photos, or photos contributed under a CONTRIBUTING.md data-license). Arden Cogar Jr.'s archive specifically requires direct outreach for permission before bulk backfill — captured as a v1.1+ blocker, not a v1 task.

### Governance and dispute handling

"Canonical record" is a claim, and claims need backing. v1 ships:
- **Public CONTRIBUTING.md governance section** — who decides identity merges (the operator for v1; aspirationally a federation-elected curator board for v2+). What happens when two sources disagree about a result (the disputes go to a `disputes/` directory in the repo, with a one-page-per-dispute markdown audit trail).
- **Dispute resolution process** — a federation or athlete can file an issue with the `dispute` template (already specced in M9 community surfaces). The operator's response is logged publicly, in the issue and in the repo.
- **Visible curator identity** — the README states who runs MNEMEX (you, by name) and how to contact you. No anonymous-canonical-record claim.

This section addresses Codex's "trust moat" finding: canonical status is institutional, not engineering. The governance is light for v1 (single operator, manual disputes) but VISIBLE.

## Appendix A: ALA / CANLOG Newsletter Format Notes

Reference: ALA September 2025 newsletter (6 pages, fixture at `tests/fixtures/newsletters/ala-2025-09.pdf`).

### Glossary

- **ALA** — American Lumberjack Association. Hosts and sanctions show events across the US/Canadian West Coast and beyond. Publishes monthly newsletters with results.
- **CANLOG** — Canadian logging-sports association (analogous to ALA, Canadian-side).
- **AWFC** — Association of Western Forestry Clubs. Organizes the college conclave circuit; member schools include Univ Montana, Univ Idaho, FVCC, Colorado State, etc.
- **M/I** — Masters / Intermediate division (used by ALA and college events; below Open, above Novice).
- **N/I** — Novice / Intermediate division (used by ALA; entry-level competitive tier).
- **OP** — Obstacle Pole event (one of the time-scored speed events).
- **HHH / VHH** — Horizontal / Vertical Hardhit (hit-counted axe events).
- **HS / VS / HSC / VSC** — Horizontal / Vertical Speed Chop (time-scored axe events; HSC and VSC are the same as HS and VS in different host conventions).
- **DQ taxonomy** — see `DQReason` enum in the schema. Mapped from per-source free-text strings via per-source mapping tables.

### Document structure

- **Page 1:** Front matter — board contacts, congratulations notes that often crosslink athletes to STIHL pro circuit ("Erin LaVoie set new STIHL underhand record"). Useful as identity-dedup hints, not as score data.
- **Page 2:** Classifieds — irrelevant to ingest. Parser should detect and skip.
- **Page 3:** Show calendar — annual table of every sanctioned show with date + location + ALA-sanctioned marker. **This is itself a high-value ingest target.**
- **Pages 4-N:** Show results, organized as `[Show Name] → [Event header] → [list of name+result rows]`. Multi-column page layout.

### Score format variability within a single newsletter

Observed in Sept 2025:
- Time-scored events with two-decimal precision (`124.00`, `9.005`): Springboard, Stock Saw, Underhand, Single Buck, Hot Saw, OP, Choker, Standing, Horizontal, Team Sawing, Jack & Jill, Double Buck.
- Place-only events with no times (`1, 2, 3, 4, 5, 6`): Axe Throw, Log Roll. Schema needs `score_type=PLACE_ONLY`.
- Mixed precision: some entries `27.25`, others `27.250`, others `27`. Parser normalizes to floats.
- DNF as a literal string in a numeric column. Maps to `dq_reason=DNF` with `final_score=NULL`.

### Athlete name canonicalization challenges (intra-document)

- **Spelling variants of the same person** appear in the same newsletter. Examples: "Mike Forester" / "Michael Forrester"; "Angus Nickolson" / "Angus Nicholson"; "Mike Johnson" / "Mike johnson" / "Mike Johsnson" (typo).
- **First-name-only entries** are common: "Frankie", "Evvy", "Kailee", "Brenda", "Bri", "Dwight". These need contextual disambiguation in the identity-review step (gender of event, partner names in pair events, division, prior-newsletter appearances).
- **Initials-only entries** appear: "B Hoffman", "Spenser Stenmark" (full but uncommon spelling).
- **Mixed casing** (e.g., "martha nicholas") is meaningless and should be normalized to title-case during ingest, but the original spelling is preserved in `name_as_recorded`.

### Wood metadata

- The Sept 2025 newsletter does NOT publish wood species or diameter per event.
- Other ALA / CANLOG newsletters reportedly do (per user). Parser must handle either case.
- When wood metadata is missing: emit row with `wood_species=NULL, wood_diameter_mm=NULL, wood_quality=NULL, wood_metadata_source='not_published'`.
- When wood metadata is present (e.g., "12in pine block, quality 7"): parse and populate the fields normally.

### Division taxonomy observed

`Open`, `M/I` (Masters/Intermediate), `N/I` (Novice/Intermediate), `Women`, `Men`. Division is sometimes explicit in the event header ("M/I Single Buck", "Women's Underhand"), sometimes implicit (event header says only "Single", with the listing context implying Open Men). Parser must infer when implicit, surface to review when ambiguous.

### Cross-document signals

Front-matter prose like "Erin LaVoie broke her prior underhand record at US STIHL Championships" is unstructured but valuable: it's an explicit identity-and-event link between two ingest sources (ALA newsletter ↔ STIHL site). Parser should extract these as `cross_source_hint` records in the identity service, used to bias future fuzzy matches.

## Reviewer Concerns

The spec-review loop ran for three iterations (the cap per the office-hours skill). Quality scores progressed 6.5 → 7.0 → 7.0. Iteration 3's four blockers (SB collision, Tier 2 contradiction with time-scored variants, identity-link false-positive policy, pending-queue storage spec) are fixed in this revision.

The following clarifications and feasibility caveats remain open. They are NOT blockers to starting Milestone 0, but they're load-bearing for downstream milestones and should be addressed in a follow-up commit during early implementation.

### Open clarifications (resolve during Milestone 0-1)

1. **Serverless deployment topology vs CLI-first v1.** The Distribution Plan still mentions Cloudflare Workers / Vercel for ingest, but the milestones describe CLI-only flows. Decision needed: either fully defer all serverless work to v2 (rewriting Distribution Plan accordingly) or add an Operational Concerns subsection on how a serverless function pushes commits to the `data/` branch (GitHub API token, queue, retry).

2. **Cost-ledger crash recovery.** Current spec uses `RESERVED` rows in the ledger. If a process crashes between `reserve()` and `finalize()/cancel()`, the reservation persists and silently consumes budget. Needs a TTL (recommended 1 hour) with reclaim-on-read at next ledger access, OR a process PID with reclaim-if-dead semantics.

3. **Imputed-quality write semantics.** Premise 7 says imputation happens with provenance flag; the projection code does default-to-5 imputation without writing back to the store. Decision needed: imputation at ingest time (writes 5 + flag=True to the store) or only at projection time (store stays NULL, projection emits 5). Recommend the latter for store cleanliness.

4. **Concurrent-ingest serialization.** The five ingest CLI subcommands (`stihl`, `college`, `newsletter`, `image`, `manual`) can in principle run simultaneously. Current spec only locks the review queue. Needs an explicit policy: either ingest commands also serialize on a shared lock, or rely on `O_APPEND` atomicity for `pending.jsonl`. Note: `O_APPEND` is not atomic on Windows; the cross-platform locking already specified in `mnemex/store.py` should be applied to ingest writes too.

5. **Secrets recovery.** No documented path for recovering a lost `.env` (only key revocation/re-issuance). Recommend a one-line addition: "key recovery requires re-issuing from provider console; previous key is revocable but not recoverable."

6. **Redaction scope for cost-ledger and `extraction_notes`.** Redaction currently covers `competitor.name_as_recorded` only. `extraction_notes` may contain raw LLM responses including names. Decision needed: either redact `extraction_notes` for affected rows during identity redaction, or document that ledger/notes never cross-reference athlete identity.

7. **`identity_resolution_required=true` semantics.** Field exists on `CanonicalRow`; behaviour not specified. Should be: if `True`, the row remains in `pending` until manually resolved and never reaches `committed`.

8. **`ExtractionStatus.NEEDS_INPUT` vs `identity_resolution_required`.** The two are conceptually overlapping. Recommend either remove `NEEDS_INPUT` (use the boolean flag instead) or differentiate explicitly: `NEEDS_INPUT` means "needs source operator to clarify the original sheet" (e.g., the photo is too blurry to read the name); `identity_resolution_required` means "names are ambiguous within MNEMEX's existing canonical athletes."

9. **`PLACE_ONLY` semantics in `final_score`.** Schema invariant 4 says `final_score = place`. But place is inverse-ordered (1 is best). Mixing this in `final_score` without a flag means downstream analytics that compute "lowest score = winner" (correct for time/hits) will produce wrong results for `PLACE_ONLY`. Recommend either add a derived `lower_is_better: bool` field, or never set `final_score` for `PLACE_ONLY` (use `place` only).

10. **`source` format collision on re-extraction.** Format is `image:<sha256>`. Re-extracting the same image after a prompt change produces two pending rows with the same `source`. Recommend extending to `image:<sha256>:v<prompt_version>` or adding an extraction-attempt counter.

11. **CLI subcommand naming.** Distribution Plan says `strathmark-export`; the integration mechanisms section uses `mnemex export strathmark`. Decision needed — pick one.

12. **Versioned prompt templates format.** Spec references `mnemex/ingest/prompts/college_excel/v1.json` (system + user templates + JSON Schema for response). No example. Punt to first prompt-engineering iteration in Milestone 3, or document the format upfront. Recommend the latter for stability.

13. **`Division` enum mixes gender and skill levels.** `Division.MEN` / `Division.WOMEN` are gender-segregated divisions; `Division.OPEN` / `Division.MASTERS_INTERMEDIATE` / `Division.NOVICE_INTERMEDIATE` are skill levels. The schema also has a separate `gender` field. Decision needed: split `Division` into a pure-skill enum and use the existing `gender` field, OR document explicitly that `Division.MEN`/`WOMEN` are gender-segregated open divisions and that the `gender` field is redundant for those rows.

14. **`tier3.py` ships in the repo tree but Milestone 6 doesn't say it ships.** Either drop `tier3.py` from the repo tree (it's not implemented), or specify Milestone 6 includes the empty `NotImplementedError("v2+")` stub.

15. **Tier 2 feature-flag mechanism.** Mechanism unspecified. Recommend an env var (`MNEMEX_ENABLE_TIER2=1`) checked at adapter load, with the default off. Activates when STRATHMARK 0.5 ships AND the operator opts in.

### Feasibility caveats (price into expectations)

16. **Vision-LLM accuracy targets.** ≥ 95% calibration / ≥ 90% ship gate is plausible for **numeric fields** on STIHL HTML (which we already scrape deterministically) and the cleaner college Excel host formats, but may be aspirational for newsletter PDFs and image scans of variable quality. The 90% ship gate is the realistic v1 floor; the 95% target should be treated as a stretch goal, not a contract.

17. **Calendar timeline at solo weekend pace.** 38-57 focused-work days at solo weekend pace (~half a day per weekend) translates to ~4-6 calendar months. Doc says "8-12 weekends" assuming full-weekend cadence. Reality is likely longer. Not a fix; just don't plan downstream milestones (e.g., STRATHEX integration) on the optimistic end.

18. **First-name-only entries are never auto-link tractable.** The new identity-matching policy (block 4) correctly addresses this: first-name-only / initials-only entries always enter the review queue with no automatic proposal. Confirmed; not a bug.

19. **Cost calibration during Milestone 3.** USD 20/day default ceiling may be tight during prompt-engineering iteration. Recommend a `--calibration-mode` flag that raises the ceiling to USD 100/day with an audit-log entry. Operator opts in explicitly.

20. **STRATHMARK-side dependencies.** `--merge-corrections` and `--split-corrections` flags on STRATHMARK's `import_legacy.py` are STRATHMARK-side work, not yet in v0.4.1. They are required for MNEMEX identity-correction propagation. Either coordinate a STRATHMARK 0.4.2 release in service of MNEMEX v1, or ship MNEMEX v1 without merge-propagation (document as v1.1 once STRATHMARK 0.4.2 lands). Recommend the former — the patch is small.

21. **STRATHMARK token-vocabulary expansion.** The new `STRATHMARK_DISCIPLINE_MAP` introduces tokens (`STB`, `SBUCK`, `DBUCK`, `JJ`, `OP`, `PSAW`, `HSAW`, `SSAW`, `SPR1`, `SPR2`) that STRATHMARK 0.4.1 has never seen. This is also a coordinated STRATHMARK release item: either ship a STRATHMARK 0.4.2 that registers the expanded vocabulary, OR restrict v1 Tier 1 export to {UNDERHAND, STANDING_BLOCK} only (the legacy vocabulary STRATHMARK already accepts) and bring the rest online with the STRATHMARK release. Recommend the latter for v1 to keep STRATHMARK changes minimal.

22. **LibreOffice on Windows.** User's primary environment is Windows. CI installs `libreoffice` via apt; local dev needs the Windows MSI. The subprocess invocation differs (`& "C:\Program Files\LibreOffice\program\soffice.exe" --headless ...`). Recommend `mnemex/ingest/college_excel.py` does OS detection and uses the right binary path; documented in CONTRIBUTING.md.

These 22 items are tracked in the project's TODOS.md (to be created at Milestone 0) and resolved during early implementation. None block starting work; all should be visible to anyone picking up the doc cold.
