# MNEMEX `data/` branch

This README is the only file on the `main` branch under `data/`. The
actual snapshots live on the orphan `data` branch, generated by the
nightly JSONL export workflow.

## What's on the `data` branch

```
data (orphan branch)
.
|-- README.md                      (this file, mirrored from main)
|-- canonical-2026-05-04.jsonl     (rolling 7-day or full snapshot)
|-- canonical-2026-05-05.jsonl
|-- canonical-2026-05-06.jsonl
...
```

Each JSONL file is one row per canonical chopping result, with the
schema of the `canonical_chopping_results` view. The view excludes
provisional rows and superseded rows; only the canonical record of each
event-discipline-competitor is exported.

## Snapshot schedule

`.github/workflows/jsonl-export.yml` runs daily at 07:00 UTC:

- **Sundays**: full snapshot. Every canonical row, all of history.
- **Other days**: rolling 7-day window. Only rows where
  `last_modified_at > now - 7 days` are exported. This catches new
  ingests, reconciliation updates, and STRATHEX finalizations from the
  past week.
- **Manual `workflow_dispatch`**: full snapshot.

Each run commits one new file to the `data` branch with the message
`data: snapshot YYYY-MM-DD (full|rolling)`.

## How to consume the snapshots

Standalone use case (no STRATHEX or STRATHMARK):

```bash
# Clone just the data branch as a shallow checkout
git clone --branch data --single-branch --depth 1 \
    https://github.com/SquirmyWormy275/mnemex.git mnemex-data

cd mnemex-data
ls -lh canonical-*.jsonl
```

Then process the JSONL with whatever tooling you have. Each line is one
canonical chopping result as a flat JSON object. Schema:

```json
{
  "result_id": "01HXYZ...",
  "competitor_id": "01HABC...",
  "discipline": "STANDING_BLOCK",
  "event_date": "2024-09-15",
  "event_name": "US STIHL Championship 2024",
  "event_circuit": "stihl_pro",
  "division": "open",
  "governing_body": "stihl",
  "log_species": "WhitePine",
  "log_diameter_mm": 320,
  "log_length_mm": null,
  "log_specification": null,
  "final_score": 24.31,
  "score_type": "time",
  "final_score_policy": "single_run",
  "handicap_mark": null,
  "partner_competitor_id": null,
  "dq_flag": false,
  "dq_reason": null,
  "no_time_flag": false,
  "scratch_flag": false,
  "runs": null,
  "last_modified_at": "2026-05-04T07:00:00Z"
}
```

Provenance fields (source_type, source_id, ingestion_run_id, provisional,
canonical_id, reconciled_at, source_native_id) are NOT in the snapshot.
The `canonical_chopping_results` view drops them. Consumers don't need
to reason about MNEMEX's reconciliation machinery; they get the
canonical record.

For STRATHMARK and other ecosystem consumers, the live database is the
source of truth (queried via `strathmark_read` JWT against the view).
The JSONL snapshots exist for:

- Standalone archive use (someone running MNEMEX as a data product
  without the rest of STRATHEX)
- Disaster-recovery and audit (the `data` branch git history is an
  immutable record of what MNEMEX has known and when)
- Bulk download for federations and analysts who want a full export to
  feed their own tooling

## Why an orphan branch instead of `main/data/`

Three reasons:

1. **`main` stays small.** The repo's source code is ~5 MB; canonical
   data could grow to GB. We don't want a `git clone --depth 1` of main
   to pull data.
2. **Snapshot history is independent.** The `data` branch's history is
   "what data has MNEMEX recorded over time," distinct from "what code
   changes has MNEMEX shipped." Mixing them confuses both audiences.
3. **Rewriting is local.** If a snapshot is later found to be wrong
   (corrupt extract, premature publish), it can be removed from the
   `data` branch without affecting `main`'s commit history.

## What the `data` branch is NOT

- It is NOT a backup of the production Supabase project. Supabase's own
  point-in-time recovery is the backup. The `data` branch is a
  human-readable archive, not a database snapshot.
- It is NOT the canonical store. The canonical store is Supabase. If the
  `data` branch and Supabase disagree, Supabase wins.
- It is NOT the sync target for STRATHMARK. STRATHMARK reads Supabase
  directly via the `canonical_chopping_results` view (PR 3).
